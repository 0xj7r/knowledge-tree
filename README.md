# knowledge-tree

A structured, append-only knowledge base that survives LLM context windows.

## The problem

Agentic and LLM-assisted work loses context on every session boundary. Each new conversation starts from zero, rediscovers decisions, and contradicts earlier conclusions. The usual response is to pile everything into a single `CLAUDE.md` or `README.md`, which quickly becomes a chronological dumping ground nobody can navigate. Decisions get re-litigated, architecture drifts from documentation, and the assistant cheerfully hallucinates answers that were settled two weeks ago.

This skill fixes that by treating project knowledge as a navigable tree of small, frontmatter-annotated markdown nodes, each with one stable purpose, a lifecycle, and an auto-regenerated index.

## The approach

Knowledge lives under `docs/knowledge/` as a tree. Every node is a markdown file with YAML frontmatter declaring `title`, `status` (`draft` / `approved` / `stale`), `last_updated`, and `related` cross-links. Updates are appended in place with timestamps. Content is never silently rewritten, only marked superseded. Each subdirectory carries a `README.md` index regenerated by parsing frontmatter from its siblings, so any LLM or human can orient themselves in one read.

The result is a living project memory that accumulates across sessions instead of evaporating at each context boundary.

## Tree structure

```
docs/knowledge/
├── README.md                   # top-level index (regenerated)
├── architecture/               # one file per architectural unit
│   ├── README.md
│   └── <component>.md
├── decisions/                  # append-only log
│   ├── README.md
│   └── YYYY-MM-DD-<topic>.md
├── integrations/               # one file per vendor/service
│   ├── README.md
│   └── <name>.md
├── contracts/                  # schemas, envelopes, API shapes
│   ├── README.md
│   └── <contract>.md
├── open-questions/             # running list with status
│   └── README.md
└── glossary.md                 # canonical terminology
```

Subdirectories are created lazily: no empty `contracts/` until there is a contract to file in it.

## How it works

### Node template

Every node begins with frontmatter:

```yaml
---
title: Vector store selection
status: approved
last_updated: 2026-04-16
related: [architecture/memory-core.md]
---
```

Body sections depend on the node type (architecture, decision, integration, contract, open-question). The `templates/` directory ships canonical scaffolds for each.

### Adding knowledge

When the user signs off on a design choice or says "document this", the skill searches the tree first, then either edits an existing node in place or creates a new one from a template. Substantive decisions also get a standalone entry under `decisions/YYYY-MM-DD-<slug>.md`, which is append-only by construction. If the decision resolves a question listed in `open-questions/`, that entry is updated with a link rather than deleted.

### Updating knowledge

Existing nodes are re-read before editing (stale in-session memory is not trusted). Changes are appended. Content that is invalidated is marked with a superseded note (date plus reason, per the convention in `SKILL.md`) rather than removed, so the historical reasoning remains legible. Material changes to an `approved` node flip its status back to `draft` until the user signs off again.

### Index regeneration

After any add or remove in a subdirectory, its `README.md` is regenerated as a table by parsing frontmatter from every sibling `.md` file:

```markdown
| Title | Status | Last updated | Link |
|---|---|---|---|
| Memory core | approved | 2026-04-16 | [memory-core.md](memory-core.md) |
| Ingest pipeline | draft | 2026-04-12 | [ingest.md](ingest.md) |
```

An un-indexed node is effectively lost, so the regen step is non-negotiable.

## Install

As a Claude Code skill:

```bash
git clone https://github.com/0xj7r/knowledge-tree.git \
  ~/.claude/skills/knowledge-tree
```

Claude Code auto-discovers skills under `~/.claude/skills/`. Invoke it by asking Claude to "add this to the knowledge base", "document this decision", or "what do we know about X", or let the skill trigger itself when you sign off on a design choice.

Alternatively, vendor the skill into a specific project under `.claude/skills/project-knowledge-tree/` to keep it repo-scoped.

## Design principles

**Never silently rewrite.** If prior content is wrong, mark it superseded with a date. History is load-bearing; reasoning that has been deleted cannot be audited.

**Status flips, not deletes.** When reality contradicts an `approved` node, the node flips to `stale` and a new version is drafted. The old content stays visible until the replacement is signed off.

**One purpose per node.** Architecture nodes describe current state. Decision nodes describe what changed and why. Conflating the two produces documents that are useful to nobody.

**Frontmatter is the API.** Indexes are regenerated by parsing frontmatter, not by guessing. If the frontmatter is missing or malformed, the node is invisible to the rest of the tree. This is the feature, not a bug.

**Re-read before editing.** In-session memory decays under context pressure. Every edit starts by re-reading the target node, because the Edit tool fails silently on stale context.

**Search before creating.** Duplicate nodes fragment knowledge and defeat the whole point. Grep the tree before assuming something is new.
